# ADR-004: 引入 Workspace Layer 作为人机共享记忆

**Status:** Accepted
**Date:** 2026-03-13
**Context:** v02 需要解决跨会话上下文丢失和数据不可见的问题

## 问题

v01 所有数据存储在 SQLite 中，用户无法直接查看阅读进度、笔记、分组等信息。每次新会话 AI 不知道用户之前在做什么。

## 选项

### 选项 A：纯数据库 + MCP 查询
- 所有状态存 DB，通过 MCP 工具查询
- 优点：单一数据源，无一致性问题
- 缺点：用户必须通过 AI 才能看到信息，无法在 IDE 文件树中浏览

### 选项 B：Workspace Layer（markdown 文件 + 数据库）
- 数据库是真相源，Workspace 文件是"投影"
- 优点：人可读、AI 可读、IDE 可浏览、支持手动编辑
- 缺点：需要维护 DB ↔ 文件一致性

### 选项 C：纯文件系统（无数据库扩展）
- 状态只存在文件中
- 优点：最简单
- 缺点：无法结构化查询，性能差，与 v01 架构不兼容

## 决策

**选择选项 B**。原因：
1. 用户明确要求"打开文件夹就能看到"
2. 数据库保证查询性能和数据完整性
3. 文件是从 DB 生成的"投影"，丢失可重建
4. AI 读取文件恢复上下文，比读 DB 更自然（markdown 就是上下文）

## 关键设计决策

- `paper_groups` 表名而非 `collections`（后者已被采集记录占用）
- Journal 采用追加模式（唯一例外），其他文件全部幂等重建
- Workspace 功能可选——不初始化不影响 v01 功能

## 后果

- 需要 WorkspaceManager 服务维护文件
- 每个写操作有额外的文件 I/O 开销（< 100ms）
- 需要 sync 机制处理手动编辑导致的不一致
